Selecting a reference architecture

Ondot cluster architecture consists of two business application layers (FIServer and ISAgent), RabbitMQ-based communication layer, and Redis-based data cache layer, and data access layer.

Within Site Single Host

This model is a most basic deployment in which all components run in one host.


One can ask why Redis runs in this model given the fact that there is only one component instance.
The answer for the question is that when FIServer or ISAgent get restarted due to unknown reasons,
it can use states stored in Redis so that a service disruption can get reduced.
Read More


Within-Site Active-Standby

This model has already supported in Ondot’s previous release from an external perspective.


This model is intended for customers who:
• the external LB or Authorization Switch is not capable of doing load balancing.
• A customer wants to upgrade to a clustering-supported release from a older release in HA setup.


Within-Site Active-Active with two hosts

This model consists of two hosts: physical or virtual hosts. Figure 3 2 Host Active-Active depicts what kinds of components run in each of hosts.


This model is intended for customers who:
• External LB and Authorization Switch can support Load Balancing
• want to have zero service downtime when a component or a host fails


Within-Site Active-Active with four hosts

This model supports higher scale and availability than the model of active-active with two hosts.


Host Active-Active depicts what kind of components run in each of four hosts. This model is intended for customers who:
• External LB and Authorization Switch can support Load Balancing
• want to have zero service downtime when a component or a host fails
• want a higher scale for current or future load


Cross-Site Active-StandBy

This model supports cross-site redundancy. This model requires DB replication across sites. Read More


The deployment model within a site can be anyone of within-site models; Active-Standby, two host active-active, and four host active-active.


Cross-Site Active-Active

This model supports active services across sites. It requires a low delay communication between sites.


In order to make clustering solution more reliable, Redis-based caching layer is not shared between sites.
Components across sites communicates using RabbitMQ Federation
